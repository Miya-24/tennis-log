# 🎾 tennis-log

テニスの **練習・試合・フィジカル** を  
**判断として管理するためのログ用リポジトリ**。

目的は反省でも記録でもなく、  
**「次の判断を軽くすること」**。

---

## 🎯 このリポジトリの目的

- 練習・試合・トレーニングの判断を **忘れない**
- 感覚的なプレーを **再利用できる形で残す**
- 技術・戦術・フィジカルを **混ぜずに整理する**
- 「どう感じたか」ではなく  
  **「次に何を採用するか／捨てるか」** を明確にする

---

## 🧠 基本思想

- GitHubは **反省を書く場所ではない**
- Issue ＝ 一次素材
- Markdown ＝ 判断の確定
- 履歴（diff）＝ 成長の可視化

失敗という概念はない。  
あるのは **採用／調整／却下**だけ。

---

## 🧱 レイヤー構造

.  
├─ docs/ # 不変のルール・思想  
├─ templates/ # 書式（Issue / Log）  
├─ logs/ # 判断が確定したログ  
├─ meta/ # 調整履歴（要約）  
└─ Issues # すべての一次ログ  



---

## 📝 運用ルール（重要）

### ① Issue（すべての入口）

- **1練習 / 1試合 / 1トレーニング = 1 Issue**
- 試合・練習・筋トレ・コンディショニングすべて Issue に書く
- 主観OK／雑OK／未整理OK
- テーマは **必ず1つ**
- 判断はしない（素材として書く）

👉 Issue は **下書き**。

---

### ② Label（文脈を決める）

#### 文脈（必須）
- `practice`：練習・練習試合・課題確認
- `match`：勝ち負けがかかる試合
- `physical`：筋トレ・コンディショニング

#### 種目（該当時）
- `doubles`
- `singles`

#### 状態（任意）
- `good`：機能した／再現できた
- `problem`：明確な課題あり
- `experiment`：試しただけ（評価保留）
- `injury-check`：違和感・ケガ関連

👉 ラベルを見るだけで  
**「誰の視点を呼ぶか」** が分かる設計。

---

### ③ Markdown（判断の確定）

以下の場合のみ、Issue を md に昇格させる。

- `match` が付いている
- `problem` / `injury-check` が付いた
- 同じテーマが2回以上出た
- 次回も使う判断だと感じた

昇格先：
- 試合 → `logs/*_match_*.md`
- 練習・筋トレ → `logs/*_practice_*.md`

👉 md は **判断の原本**。  
👉 迷いが残る内容は md にしない。

---

### ④ meta（現在地）

`meta/adjustment_history.md` には  
**「なぜその判断になったか」だけを書く**。

- 詳細は logs にある
- meta は **今の方針を確認する場所**

普段読むのは **meta だけ**。

---

## 🧠 コーチ／トレーナー視点（判断軸）

### 技術（橘）
- 10回できるか
- フォーム修正の前に  
  準備・打点・高さ・深さで解決できるか

### 戦術
- ダブルス：佐倉（待つ）／鷹宮（仕掛け）
- シングルス：白石（流れ操作・決断）

### フィジカル
- 原田：今やる意味があるか
- 高瀬：明日ラクか
- 高橋：続くか

👉 **NGが出たら即中止**。

---

## ❌ このリポジトリでやらないこと

- 感情だけの反省
- 数値管理・統計
- 根性論・我慢
- 過去ログの読み返し前提運用

---

## ✅ 正しい使い方（要約）

- 普段：Issueを書く → close
- 判断が必要：Issue → md
- 方針が固まったら：meta に1行
- 迷ったら：logs を1つだけ見る

---

## 🏁 最後に

このリポジトリは  
**うまくなった証拠を集める場所ではない**。

**次の判断を軽くするための外部記憶**。

今日の Issue 1つで、
